### Task #12: donglicao 功能修复 🔧

- **严重度**: 🟡 High（影响高级功能）

#### 根因分析

##### 深度思考模式
```
调用链：
POST /v1/chat/completions (thinking: true)
  ↓
routes/chat_endpoints.py - 解析请求 ✅
  ↓
routes/chat_handler.py - maybe_thinking_response()
  ↓
routes/chat_support.py - thinking_route()
  ↓
THINKING_BACKENDS 选择:
  1. or_deepseek_r1 → ❌ 401 Unauthorized (OpenRouter key 失效)
  2. kimi_thinking → ❌ Connection Refused
  3. scnet_ds_pro → ✅ 可用
  ↓
问题：前两个后端失败后，仍返回 500（而非继续尝试）
```

1. OpenRouter API key 已过期/失效
2. `kimi_thinking` 服务不可达
3. 错误传播链不完整，导致 500 而非优雅降级

2. ✅ **错误处理增强**: 401 特殊日志
3. ⚠️ **仍有 500 错误**: 需要更深入调试

## 架构亮点 ✨

### 1. **智能路由系统**
```
用户请求
  ↓
intent 检测（工具/视觉/思考/普通）
  ↓
候选后端选择（优先级 + 健康度）
  ↓
协议适配（OpenAI ⟷ Anthropic）
  ↓
流式响应（SSE 格式统一）
```

**优点**:
- 灵活的后端切换
- 协议无关的上层逻辑
- 易于扩展新后端

### 2. **协议适配层**
- `opencode_protocol_adapter.py` - OpenAI ⟷ Anthropic
- `anthropic_converter.py` - 消息格式转换
- `vision_handler.py` - 多模态适配

**优点**:
- 解耦上层业务和底层协议
- 统一的接口抽象
- 易于支持新协议

### 3. **分层依赖注入**
```python
# routes/chat_endpoints.py
inject_deps(
    handle_chat=...,
    anthropic_native_stream=...,
    model_id=...,
)
```

**优点**:
- 模块解耦
- 易于测试和mock
- 清晰的依赖关系

**缺点**:
- 学习曲线略陡
- 运行时依赖注入增加复杂度
