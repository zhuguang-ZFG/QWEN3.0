# P0.1 ESP32 Motion Executor Contract — Implementation Plan

> Date: 2026-05-25
> Parent: `docs/superpowers/plans/2026-05-25-productivity-infrastructure-review.md`
> Scope: ESP32 firmware + Device Gateway Server + fake-U8 smoke

## Goal

Every motion task dispatched through the Device Gateway produces one of:
`accepted` → `running` → `progress` → `done` | `failed`
with explicit error codes. Default boards must fail loudly, not silently no-op.

## Current State (from PROD-001, PROD-002)

|------|-------|

## Implementation Slices

### Slice 1: Server-Side Error Codes + Protocol Contract

**Files:**
- Modify: `device_gateway/protocol_families.py`
- Modify: `device_gateway/protocol.py`
- Test: `tests/test_device_gateway_protocol_families.py`

**Steps:**
- [ ] Add `MotionErrorCode` enum to protocol: `E_UNSUPPORTED_CAPABILITY`, `E_MISSING_PATH`,
  `E_BAD_PARAMS`, `E_U1_UNAVAILABLE`, `E_DEVICE_UPDATING`, `E_EXECUTION_FAILED`.
- [ ] Add `motion_failure_event()` helper that constructs a standards-compliant
  failure frame from device_id, task_id, error_code, and reason.
- [ ] Add `validate_motion_task_lifecycle()` that asserts every task gets at
  least one terminal event (done or failed).
- [ ] Extend fake-U8 tool to send explicit failure events by error code.
- [ ] Verify: fake-U8 failure smoke returns correct error code in Server task state.

### Slice 2: Device Gateway Path Validation Before Dispatch

**Files:**
- Modify: `device_gateway/tasks.py`
- Create: `device_gateway/path_validator.py`
- Test: `tests/test_device_gateway_path_validator.py`

**Steps:**
- [ ] Add `validate_run_path_params()` hardening: reject empty paths, oversized
  paths, out-of-bounds points before task creation.
- [ ] Add capability → path requirement mapping so unsupported capability
  returns deterministic `E_UNSUPPORTED_CAPABILITY` on the Server side.
- [ ] Wire `validate_run_path_params()` into `project_to_motion_task()` so
  invalid tasks fail at creation time with structured errors.
- [ ] Verify: bad path/unsupported capability/capability mismatch all produce
  specific error codes in task state without reaching fake-U8.

### Slice 3: ESP32 Firmware — Default Board Fail-Loud

**Files (in esp32S_XYZ submodule):**
- Modify: `firmware/u8-xiaozhi/main/boards/common/board.cc`
- Modify: `firmware/u8-xiaozhi/main/boards/common/board.h`

**Steps:**
- [ ] Replace empty `HandleMotionTaskJson()` default with a fail-loud
  implementation that emits `failed` + `E_UNSUPPORTED_BOARD`.
- [ ] Declare `virtual bool SupportsMotionTask(const char* capability)` so
  boards can opt-in explicitly.
- [ ] Add `SendMotionEvent(task_id, phase, error_code, reason)` helper to the
  common board Uplink interface.
- [ ] Verify: fake-U8 or unsupported board smoke produces a Server-visible
  failure event within one smoke run.

### Slice 4: ESP32 Firmware — Zhuguang Board Failure Hardening

## Exit Criteria

A missing-path or unsupported-capability motion task is visible as a
structured failure event in Server task state within one smoke run. A valid
task produces the full accepted→running→done timeline. No task is silently
lost.
