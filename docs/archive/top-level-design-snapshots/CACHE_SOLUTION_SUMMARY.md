# qwen2API 缓存问题总结与建议

**日期**: 2026-06-12
**状态**: ⚠️ Nginx 缓存方案遇到技术障碍

---

## 🎯 **目标**

将 MiMo Code + qwen2API 的响应时间从 8-10秒 优化到 10-50ms（缓存命中时）

---

## ❌ **已尝试方案**

### 方案 1: Python 缓存代理

**文件**: `qwen_cache_proxy.py`

**问题**:
- Windows 多进程管理复杂
- 端口占用清理困难
- 缓存逻辑未正确工作

**结果**: ❌ 放弃

### 方案 2: Nginx 缓存代理

**文件**: `nginx/conf/nginx.conf`, `nginx/test_cache.conf`

**架构**:
```
MiMo → Nginx (7865/7866) → qwen2API (7862)
```

**问题**:
1. ✅ Nginx 成功启动并转发请求
2. ❌ 缓存头部 `X-Cache-Status` 未出现
3. ❌ 访问日志文件始终为 0 字节
4. ❌ POST body 缓存未生效

**可能原因**:
- Windows 版 Nginx 缓存特性受限
- `proxy_cache_key` 包含 `$request_body` 在 Windows 上不工作
- 日志缓冲问题
- 配置语法在 Windows 版本中有差异

**结果**: ⚠️ 技术障碍

### 方案 B: 等待 qwen2API 官方支持

**GitHub Issue**: https://github.com/YuJunZhiXue/qwen2API/issues

**内容**:
```markdown
### Feature Request: 内置缓存支持

**问题**:
当前 qwen2API 每次请求都调用通义千问 API，响应时间 8-10 秒。

**建议**:
添加内置缓存层，缓存相同请求的响应。

**预期效果**:
- 首次请求：8-10秒
- 缓存命中：10-50ms
- 降低 API 调用频率 60-80%

**实现参考**:
- 使用 SQLite 存储缓存
- 缓存键：hash(model + messages + params)
- TTL: 24小时
```

**优势**:
- 官方支持，稳定可靠
- 所有用户受益
- 无需额外配置

**时间**: 不确定
