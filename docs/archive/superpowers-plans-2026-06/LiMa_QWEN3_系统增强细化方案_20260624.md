## P0-1 后端预热连接池

### 现状分析

当前 `http_sync.py` 的 `call_api()` 每次调用都通过 `_build_client()` 创建新的 `httpx.Client` 上下文管理器：

```python
# http_sync.py 第 130 行
with hc._build_client(backend, timeout) as client:
    resp = client.post(cfg["url"], content=body, headers=headers)
```

每次创建新 client 意味着：DNS 解析 → TCP 三次握手 → TLS 握手，连接无法复用 keep-alive。对于 170+ 后端中 top-20 高频后端，每秒可能被调用数十次，连接开销显著。

### 实现方案

#### 步骤 1：新建 `connection_pool.py`

```python
"""后端持久连接池管理器。为 top-N 高频后端维护 httpx.Client 实例。"""
import httpx, logging, threading
from typing import Dict

_log = logging.getLogger(__name__)
_POOL_SIZE = 20
_MAX_KEEPALIVE_SECONDS = 60

class ConnectionPool:
    def __init__(self, max_size: int = _POOL_SIZE):
        self._max_size = max_size
        self._clients: Dict[str, httpx.Client] = {}
        self._lock = threading.Lock()
        self._backend_priorities: list[str] = []

    def get_client(self, backend: str, timeout: int = 60) -> httpx.Client:
        with self._lock:
            if backend in self._clients:
                return self._clients[backend]
            if len(self._clients) >= self._max_size:
                self._evict_oldest()
            client = httpx.Client(
                limits=httpx.Limits(max_keepalive_connections=5, max_connections=10,
                                    keepalive_expiry=_MAX_KEEPALIVE_SECONDS),
                timeout=httpx.Timeout(timeout), http2=True,
            )
            self._clients[backend] = client
            return client

    def _evict_oldest(self):
        candidates = [b for b in self._clients if b not in self._backend_priorities[:5]]
        if candidates:
            self._clients.pop(candidates[0]).close()

    def warmup(self, backend_configs: dict):
        for backend in self._backend_priorities[:self._max_size]:
            cfg = backend_configs.get(backend)
            if cfg:
                try:
                    self.get_client(backend, cfg.get("timeout", 60))
                    _log.info("Warmed up connection: %s", backend)
                except Exception as exc:
                    _log.warning("Warmup failed for %s: %s", backend, exc)

    def close_all(self):
        with self._lock:
            for client in self._clients.values():
                try: client.close()
                except: pass
            self._clients.clear()

_pool = ConnectionPool()
def get_pool() -> ConnectionPool: return _pool
```

#### 步骤 2：修改 `http_sync.py` 的 `call_api()`

替换 `with hc._build_client(backend, timeout) as client:` 为从连接池获取：
```python
from connection_pool import get_pool
client = get_pool().get_client(backend, timeout)
resp = client.post(cfg["url"], content=body, headers=headers)
resp.raise_for_status()
```

#### 步骤 3：在 `server_lifespan_phases.py` 添加预热阶段

```python

    async with PhaseTimer("connection_pool.warmup"):
        from connection_pool import get_pool
        from backends_registry import BACKENDS
        from health_tracker import get_scores
        scores = get_scores()
        get_pool()._backend_priorities = sorted(scores.keys(), key=lambda b: -scores.get(b, 0))[:20]
        get_pool().warmup(BACKENDS)

WARM_PHASES.append(("connection_pool.warmup", warmup_connection_pool))
```

#### 步骤 4：停机关闭

在 `server_lifespan.py` 的 `_run_shutdown_phases()` 末尾添加 `get_pool().close_all()`。

### 实现方案

#### 步骤 1：新建 `trace_context.py`

使用 `contextvars` 在异步调用间传递 trace_id，所有步骤通过 `trace_record()` 写入 SQLite：

```python
import contextvars, json, sqlite3, time, uuid
from config.sqlite_pool import get_pooled_connection

_current_trace: contextvars.ContextVar[dict | None] = contextvars.ContextVar("trace", default=None)
_DB_PATH = "data/trace_store.db"

def start_trace() -> str:
    trace_id = f"trc_{int(time.time()*1000)}_{uuid.uuid4().hex[:8]}"
    _current_trace.set({"trace_id": trace_id, "order": 0})
    trace_record("trace_start", {"trace_id": trace_id})
    return trace_id

def trace_record(step: str, data: dict | None = None, duration_ms: int = 0) -> None:
    ctx = _current_trace.get()
    if not ctx: return
    ctx["order"] += 1
    conn = get_pooled_connection(_DB_PATH)
    # ensure schema + insert
    conn.execute("INSERT INTO request_traces (trace_id, step, step_order, timestamp, data, duration_ms) VALUES (?,?,?,?,?,?)",
                 (ctx["trace_id"], step, ctx["order"], time.time(), json.dumps(data or {}), duration_ms))
    conn.commit(); conn.close()

def get_trace(trace_id: str) -> list[dict]: ...
def clear_trace(): _current_trace.set(None)
```

## P1-2 语义缓存增强

### 现状分析

`context_pipeline/cache.py` 仅做 prefix 缓存（稳定前缀哈希），不做 query-level 语义缓存。

### 实现方案

新建 `semantic_cache.py`：基于 embedding 相似度的查询结果缓存。

- **查询**：计算 query embedding → 先精确 hash 匹配 → 再语义扫描最近 500 条，余弦相似度 > 0.95 命中
- **写入**：成功请求结果写入 SQLite，TTL 1 小时，上限 10000 条
- **集成**：在 `routing_engine.py` 的 `route()` 开头检查缓存，成功请求结束后写入缓存
- **限制**：仅非流式 + 非 IDE 请求走缓存

## P2-2 多模型编排（MoE）

### 现状分析

`routing_executor.py` 的 `execute()` 是串行 fallback：选一个后端 → 失败则下一个。复杂编码任务单次调用质量有限。

### 实现方案

新建 `orchestrator.py`：三阶段编排（规划器 → 编码器 → 审查器）。

- **触发条件**：query ≥ 200 字符 + coding 场景 + complexity == "complex"
- **Stage 1**：规划器（最强后端）分析请求，生成 5 点实现策略
- **Stage 2**：编码器（top-3 后端）按策略生成代码
- **Stage 3**：审查器（第二强后端）审查，PASS 直接返回，FAIL 带反馈重试一次
- **Fallback**：编排失败自动回退到原有执行路径

修改 `routing_engine_execute_strategy.py` 的 `execute_with_strategy()` 添加编排分支。

## P3-1 至 P3-8 设备与安全增强

### P3-1 设备网关事件溯源

**现状**：`device_ledger/` 已有事件存储基础，`task_lifecycle.py` 通过 `ledger_store.append_event()` 记录事件。

**增强**：
- 扩展事件类型：`task_created`、`task_validated`、`task_dispatched`、`task_acked`、`task_completed`、`task_failed`
- 新增 `device_ledger/projection.py`：从事件流重建任务状态
- 新增 `device_ledger/replay.py`：按时间点回放
- 新增 `GET /v1/admin/tasks/{task_id}/timeline` 端点

### P3-2 设备意图理解增强

**现状**：`device_gateway/intent.py` 使用规则 + LLM 混合解析。

**增强**：
- 从 `outcome_ledger` 提取历史意图分类数据
- Fine-tune Qwen-1.5B 做本地意图分类
- 部署为本地推理服务，`intent.py` 优先调用本地模型，fallback 到规则 + LLM

### P3-3 固件远程证明

**增强**：
- 新增 `device_gateway/attestation.py`：设备连接时提交固件哈希
- 服务端比对已知固件版本白名单
- 不匹配则降级为只读模式

### P3-4 API 请求异常检测

**增强**：
- 新增 `security/anomaly_detector.py`：基于请求频率/Token 消耗/后端分布的异常检测
- 使用滑动窗口统计 + Z-Score 检测
- 异常请求记录到 `security/anomaly_events.db`

### P3-5 AI 路径优化器

**现状**：`device_gateway/path_pipeline.py` 的 `text_to_path()` 和 `svg_path_to_motion()` 是确定性的，无优化。

**增强**：
- 新增 `device_gateway/path_optimizer.py`：路径压缩（合并重复点）、平滑（B-spline 近似）、最短路径排序
- 对 `render_svg_task()` 的输出做后处理优化

### P3-6 设备预测性维护

**增强**：
- 新增 `device_gateway/maintenance.py`：从 `outcome_ledger` 提取设备任务成功率/延迟趋势
- 计算设备健康评分，低于阈值时主动通知
- 每日 cron job 生成设备健康报告

### P3-7 多设备协同绘制

**增强**：
- 新增 `device_gateway/coordinator.py`：将大幅作品分割为多个子区域
- 分配给不同设备并行绘制
- 通过 `device_ledger` 追踪协同进度

### P3-8 A/B 测试框架

**增强**：
- 新建 `ab_testing.py`：按请求哈希分流（10% 实验组 vs 90% 对照组）
- 实验组使用新路由策略/新后端
- 从 `outcome_ledger` 收集对比数据
- `GET /v1/admin/experiments` 查看实验结果
